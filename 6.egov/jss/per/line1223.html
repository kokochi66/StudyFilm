자바 스택 중심으로 해서 자기 소개하기
안녕하세요 자바 서버개발자가 되기 위해서, 언제나 발전하려 노력하는, 25살 김재원이라고 합니다.
제 기술 스택에 대해서, 간단하게만 두가지 설명드리도록 하겠습니다.
첫번째 저의 기술스택은 자바입니다. 먼저 자바 언어 자체에 대해서, 다양한 코딩 테스트 문제들을 많이 풀어봄으로써, 언어에 대한 지식을 넓혔습니다.
코딩 테스트 문제를 풀 때에도, 문제에서 필요한 객체를 만들고, Comparable의 compareTo 메소드를 재정의 하여 Collection Sort나 PriorityQueue 등 컬렉션을 자유롭게 사용할 수 있는 능력이 있습니다.
또한 참조에 의한 호출을 잘 이해하여, 특정 객체를 Map과 List에 함께 넣고, 필요한 값은 List에 추가하고, 리스트에 맞는 키값에 따라서, 하위 데이터를 추가하는 방식을 즐겨 사용합니다.

두번쨰 저의 기술스택은 스프링입니다. 스프링 4.0 버전에 대해서 기본적으로 공부를 하였고요, MVC와 IoC 패턴에 대해서 집중적으로 공부를 했습니다.
이를 토대로 트위치 팔로우 매니저라는 개인 프로젝트를 만들면서, 대용량 데이터를 처리하여, 사용자에게 편리한 서비스를 만든 경험이 있습니다.
추가로, 지난 1차 면접에서는 트랜잭션이나, AOP에 대해 아직 개념이 부족해서, 관련 질문에 대해서 대답을 잘 못했던 부분이 있었습니다.
이 부분에 대해서 지금은 정말 많이 조사하여, 트랜잭션은 어떻게 적용을 하는지, AOP는 실무에서 어떻게 적용을 할지 인턴을 하고 있는 회사의 다양한 프로젝트 설정 파일들을 보면서 공부를 많이하여, 이 부분에서도 잘 알게 되었습니다.

이런 능력들을 토대로, 라인 스튜디오 인턴십에 참가하면, 서버 개발에 대해 더 공부하여 더 좋은 서버, 안정적인 서버를 구축할 수 있는 개발자가 되고싶습니다.

자신의 장점과 단점을 이야기 해 보세요
저의 가장 큰 장점은 코드의 에러를 잘 찾아내는 능력입니다.
저는 코드에서 에러가 발생하거나, 원하지 않은 결과가 나타났을 때, 잘못된 코드로 추측되는 모든 부분에 많게는 몇십개씩 로그를 찍습니다.
이렇게하면 로그를 찍는 과정은 귀찮을 수 있으나, 정확하게 어디서 에러가 발생했는지 알 수 있고, 보다 직관적으로 어떤 원인에 의해 에러가 발생했는지 파악이 쉽기 때문에 해결하는 속도가 굉장히 빠릅니다.
그래서 비교적 남들보다 에러상황을 잘 찾고, 더 잘 해결할 수 있다고 생각합니다.

저의 단점이라기 보다, 팀 프로젝트를 진행함에 있어서 고민이되는 부분이 있습니다.
어떤 사람이 만든 코드에 대해서, 제가 이 코드는 확실하게 잘못된 부분이 있다고 판단을 내렸을 때, 저는 돌려 말하기 보다 직설적으로 잘못된 부분을 짚고, 고쳐야 한다고 이야기 하는 스타일입니다.
애매하게 말하면, 어떤 부분이 잘못되었는지 파악이 어려울 수 있고, 문제가 해결되지 않는다면 프로젝트에 잘못된 영향을 줄 수 있습니다.
하지만 그 사람에 대해 지적을 하는것은 그 사람의 자신감을 잃게 만드는 원인이 될 수 있고, 자신감을 잃고, 멘탈이 흔들리다보면 자연스럽게 일의 능률에도 영향이 갑니다.
그래서 이 부분에 대해서 어떤 방법이 무조건 옳다, 틀리다 라고 결론을 내리기가 굉장히 어려운 것 같습니다.
팀 프로젝트라는 것은 여러 사람들과 함께 힘을 합쳐야 하는 일인만큼, 원만한 인간관계와 프로젝트의 올바른 방향성을 사이에 두고, 저울질하고 고민하면서 정확한 판단을 할 수 있는 개발자가 되고 싶습니다.
개발자로써 중요하다고 생각하는 가치는 무엇인가요?
항상 최선의 방법이 무엇일지 고민하는 것이라고 생각합니다.
최선의 방법이라는 것은, 단순히 코드의 길이를 줄이는 것 부터, 시간복잡도를 계산해서 더 효율이 좋은 코드를 고민하는 것인데요
이런 마음가짐을 기본으로 갖고 개발을 해 나가는것이 올바른 개발자의 책무라고 생각합니다.
다만, 고집은 부려서는 안됩니다. 최적화를 고집하여, 코드의 길이를 줄이는 것에만 고집하다보면 유지보수에 어려움이 생길 수 있습니다.
또, 이미 끝난 문제를 계속해서 붙잡고 있으면 팀적으로도 좋지 못한 결과를 초래할 수도 있습니다.
그렇기 때문에 최적화를 기본적인 마음가짐으로 갖고있되, 상황에 따라서 올바른 방향을 판단하는 능력을 길러야 한다고 생각합니다.

+ 최적화로 인해서 유지보수가 어려워지는 사례 : 중복된 코드를 줄이기 위해서 별도의 함수를 만드는 작업에서 이런 문제가 생길 수 있을 것 같습니다.
원하는 작업 내용이 조금씩 다른 것을 단순히 코드의 유사도만 갖고 함수를 만들어서 공유했을 때, 특정 부분에 문제가 생겨 함수의 특정 부분을 수정했다가 다른 부분에도 영향을 미친다면
이는 코드의 길이만을 줄였을 뿐, 향후의 유지보수에 어려움을 주는 최적화가 됩니다. 그래서 항상 최적화와 유지보수는 동전의 앞뒷면처럼 함께 생각하면서, 명심하고 개발해야 할 것 같습니다.
인턴에 합격하게 되면 어떤 부분을 배워나가고 싶으신가요?
제 개인 프로젝트인 트위치 팔로우 매니저라는 웹 어플리케이션이 있습니다.
해당 어플에서는 API를 통해서, 대용량으로 데이터를 주고받고, 데이터를 분석하는 작업을 실행하는데, 수많은 데이터들을 실시간으로 받아와서 분석에 반영해야하기 때문에 제가 구현할 수 있는 부분에서 성능적인 한계가 보였습니다.
그래서 DB 쿼리문을 짤 때, Mybatis를 사용해서 반복문으로 한번에 여러개의 데이터를 INSERT하거나, FIND_IN_SET을 사용해서 한번에 여러개의 데이터를 삭제하는 방식을 사용해서 쿼리문 자체를 몇백개씩 날리는 부분을 해결하려했는데요.
이런 실제, 대용량 데이터를 주고받는 API의 경우에는 이런 부분을 어떻게 쿼리문을 짜서 해결하는지, 컨트롤러 구성은 어떻게 되어 있을지에 초점을 맞춰서 배워나가고 싶습니다.

트러블 슈팅에 대해서 잘못 이해한 부분
트러블 슈팅을 저는 팀 프로젝트 진행간에, 의견 충돌로 인해서 문제가 발생한 부분으로 이해하여, 잘못 답변을 했던 것 같습니다.
그래서 다른 트러블 슈팅 경험에 대해서 말씀 드리겠습니다.
나모 회사 인턴에서 유지보수 작업을 하는 중에 생긴 이슈입니다.

문제 내용은, 문제 메세지 발송으로, 본래 A 내용의 메세지가 가야하는 조건에서 B 내용의 메세지가 전송되는 오류였습니다.
해당 문제 상황을, 이전에 제가 해당 회사 다른 사이트의 유지보수 작업에서 문자 메세지 작업을 한 적이 있어, 논의 후에 제가 작업을 맡게 되었습니다.

문제의 상황 파악을 위해서, 기능이 어떤 구조로 돌아가고 있는지를 파악하였습니다.
메세지는 특정 인원에게 특정 플래그를 주어진 상태에서, 문자 메세지를 보낼 시간을 설정하면, 문자 메세지에 대한 데이터가 별도의 문자 전송 서버의 데이터 베이스로 접속하여, 전송하는 구조로 되어 있었습니다.

여기서 해당 문제가 발생한 원인을 찾았는데, 플래그를 설정하지 않고, 문자 메세지 전송 시각을 먼저 설정 후에, 플래그를 설정하면, 이미 별도의 서버로 데이터가 넘어간 상태이기 때문에, 정상적으로 문자 내용이 적용되지 않는 부분이었습니다.
이 플래그가 변경 될 때마다, 해당 서버와 커넥션을 통해 연결하여, 해당 데이터를 변경하기 위해서는, 문자 서버 데이터베이스에 들어가있는 문자 데이터의 Id값을 운영중인 서버의 데이터베이스에서 별도로 저장 할 필요가 있었습니다.
그런데 이를 위해서, SMS정보를 운영중인 서버 내에서 별도의 테이블로 계속해서 데이터를 저장하는 구조 구현에는 어려움이 있었고, 상당히 시간이 필요한 작업이었습니다.
그래서 이 부분을 다른 개발자분들과 의논 후에, 플래그값이 바뀔때마다 서버 데이터베이스에 접속해서 업데이트를 하는 방식이 아니라,
플래그를 먼저 설정하지 않으면, 문자 메세지 전송이 불가능하며, 문자 메세지 전송시에, 자바스크립트 Confirm 메세지를 출력하여, 다시한번 플래그가 설정되어있는지를 확인하도록 하여 해결했습니다.
재미있게 들었던 학교 수업과 기억나는 내용
학교 수업중에 기억이 남는 것은 모바일 프로그래밍 수업입니다.
안드로이드 스튜디오를 활용하여, 여러가지 실습 어플리케이션을 만들어보는 수업이었는데요.
여기서 가장 기억에 남는 것은 실제 영화관 홈페이지의 오픈 API를 HTTP 통신으로 가져와서, 영화 정보를 어플에서 검색하면, 데이터를 표시해주는 어플리케이션을 만든 것이었습니다.
여기서 Restful API도 처음 접하게 되었고, 어플리케이션에서 이런 API를 활용해서 HTTP 통신으로 어플리케이션을 만들 수 있다는 부분에 흥미를 갖게 되었습니다.
10년 후 어떤 개발자가 되고 싶은지
서버 개발자로써, 계속 발전해나가고 새로운 것들을 배워나감을 통해서
누군가가 이 서버에 관한 질문을 저에게 했을 때 모두 자신있게 대답할 수 있는 사람이 되고 싶습니다.
이런 정보를 전부 머릿속에 외우고 있을수는 없더라도, 적어도 이 개념은 이것에 대한 부분이며, 이 구조는 어떻게 되는지를 설명할 수 있는 사람
말 그대로 프로 페셔널하게 자신의 직업의 내용에 대해서 정확하게 알고있는 그런 프로 개발자가 되고 싶습니다.
입사한다면 어떤 일을 하고 싶은지
일단 딱 입사해서 회사에 들어간다면, 예제 프로젝트나, 서버의 코드를 볼 수 있으면 좋겠습니다.
제가 인턴 회사에 처음 입사했을 때에는 회사의 CMS 파일을 svn을 통해서 받아서, 그 CMS를 자세하게 조사해보는 경험을 했습니다.
처음에는 무엇부터 봐야할지 잘 모르겠는 부분이 있어서 헤메었지만 지금은 어느정도 프로젝트 파악을 하려면 어떻게 확인을 해야하는지를 익혔다고 생각합니다.
먼저 @Configuration 어노테이션이 사용된 자바파일이나, xml파일을 통해서 DB 설정, AOP, 인터셉터, 시큐리티 설정은 어떻게 되고 있는지, pom.xml 을 통해서 버전관리는 어떻게 되어있는지 자바 버전은 어떠한지
그 다음은 컨트롤러를 통해서 메인으로 사용되는 컨트롤러가 어떻게 되어있고, Service단과는 어떻게 연결이 되어있는지, 트랜잭션 처리는 AOP로 하는지, xml로 하는지 파악을 하고
다음은 전체적으로 VO, DAO 등을 파악하면서 명칭을 어떤 규칙으로 적용을 하고, 변수 이름은 언더바를 사용하는지, Camel 표기법을 사용하는지,
혹은 예를들면 제 인턴 회사에서는 @Autowired를 사용하지 않고, @Resource로 빈의 명칭을 명시하는 방식을 사용했는데, 이 부분에서는 어떤 규칙을 사용하는지
이렇게 회사에서 적용하고 있는 코딩 규칙들이 어떤 부분들이 있는지 제 나름대로 정리하고 알아보는 시간을 가져야한다고 생각합니다.

이런 기본적인 부분이 되어야, 프로젝트를 한다고 해도 정상적으로 해당 프로젝트에 대해서 이해를 하고 참여할 수 있다고 생각합니다.
라인, 네이버등 많이 지원을 하고있는데 그냥 계속 지원을 넣고있는건지?
지난 1차 면접에서 제가 어느 부분에서 지식이 많이 부족한지, 어떤 부분을 중심으로 공부를 해 나가야 할지 방향성을 정할 수 있었다고 생각합니다.
특히 저는 DB부분에서 지식이 부족해서 이 부분에서 계속해서 공부해나가야 한다고 생각이 들어서, DB 관련되어 좀 자세하게 공부할 수 있는 서적이나 강의들을 찾아보고 있었습니다.
이렇게 면접은 저는 굉장히 좋은 경험이라고 생각을 하고, 제가 더 좋은 개발자로 발전해 나가기 위해서 이런 경험을 계속해서 겪어야한다고 생각합니다.
그래서 계속해서 지원하고, 이렇게 함으로써 저 자신에게도 자신이 부족함을 깨닫는 시간을 가지고, 더 공부해야겠다는 동기부여를 얻는 것 같습니다.
